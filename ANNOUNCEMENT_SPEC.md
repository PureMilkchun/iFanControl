# 每周升级公告规范

## 格式模板

```markdown
## iFanControl 每周升级公告（YYYY.MM.DD – YYYY.MM.DD）

**数据统计**
- 提交次数：N 次
- 文件变动：N 个文件
- 代码变动：+N 行 / -N 行
- 版本跨度：vX.Y.Z（build N）→ vX.Y.Z（build N）

---

### 功能名称

一到两句话描述功能内容，面向用户，不暴露内部实现细节。
```

## 数据统计获取方式

在项目目录下执行：

```bash
# 提交次数
git log --since="YYYY-MM-DD" --until="YYYY-MM-DD" --oneline | wc -l

# 文件变动和代码行数
git diff --stat HEAD~$(git log --since="YYYY-MM-DD" --until="YYYY-MM-DD" --oneline | wc -l) HEAD
```

## 内容规范

### 标题

- 格式：`iFanControl 每周升级公告（起始日期 – 结束日期）`
- 日期用中文句号分隔：`2026.04.25 – 2026.05.03`

### 数据统计

- 提交次数：`git log --oneline` 的条目数
- 文件变动：`git diff --stat` 的文件数
- 代码变动：`git diff --stat` 的 insertions / deletions
- 版本跨度：起始版本 → 最新版本，包含 build 号

### 内容条目

- 每个独立功能/修复为一个小标题（`###`）
- 小标题用功能面向用户的名称，不用内部模块名
- 正文一到两句话，面向用户语言，不暴露实现细节
- 涉及公开接口、数据格式、配置项等内部实现的内容一律不写
- 修复类条目放在最后，标题统一为「Bug 修复」，内容用列表

### 叙述口径

- 只写用户可感知的变化，不写内部架构重构
- 心跳/统计相关改动统一写成「修复若干问题，提升整体稳定性」
- 性能优化写体感效果（"即时响应"、"内存降至 39MB"），不写内部机制

## 条目分类顺序

1. 性能与体验优化
2. 新功能
3. 稳定性与兼容性改进
4. Bug 修复（统一用列表）
